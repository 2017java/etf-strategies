# Inspo MVP 功能规格书（PRD）— 交叉评审报告

**评审日期**：2026-05-20
**基准文档**：`Inspo-竞品与产品价值分析-2026-05-20.md`
**被评审文档**：`Inspo-MVP功能规格书-PRD-2026-05-20.md（v1.1）`
**评审方法**：逐项对比两份文档的 P0/P1/P2 分配、功能取舍、用户场景覆盖、Non-goals 一致性、假设验证机制

---

## 📌 TL;DR 评审结论

**总体评价**：PRD 是一份质量不错的详细功能规格书，细节充实、成本模型扎实。但存在 **3 个关键结构性问题**，如果不在 V1.0 前修正，可能导致 MVP 验证目标偏离：

| # | 问题等级 | 问题 |
|---|:------:|------|
| 1 | 🔴 **高** | **核心差异化「灵感→待办」从 P0 降级到 P1** — 等于 MVP 不做自己的核心差异化 |
| 2 | 🔴 **高** | **缺少核心假设验证机制** — 竞品分析明确提出最需要验证的假设（灵感记录是否高频刚需），PRD 有 KPI 但无实验设计 |
| 3 | 🟡 **中** | **竞品响应策略缺失** — 竞品分析评估 flomo 跟进概率为"高"，但 PRD 未设计时间窗口和抢先策略 |

其余问题为 🟢 低优先级，主要是功能细节层面的对齐和澄清。

## 二、🔴 关键问题 1：核心差异化待办功能降级

### 问题描述

竞品分析第 8 节 P0 功能矩阵明确列出：

> | 灵感→待办一键生成 | 核心差异化×最小闭环 | 中 |

这是在竞品分析中与 flomo 区分的最核心功能（"flomo 的核心盲区：完全缺失行动闭环"）。竞品分析第 3.3 节更明确地将待办转化定位为 **⭐ 核心差异化**。

PRD 将其移到 P1（V1.1，即 V1.0 发布后 4-6 周），理由未显式说明。PRD 的 P0 变成了：
- 记录 + AI标签 + AI对话

**这意味着**：V1.0 发布时，Inspo 的核心体验 = 一个带 AI 自动标签的笔记工具。这和竞品分析中定位的"灵感→行动闭环"有本质差距。

### 影响评估

| 维度 | 影响 |
|------|------|
| **差异化** | V1.0 用户看不到任何"待办转化"能力，只能看到"AI标签"。flomo 也在做 AI（AI找一找、AI洞察），虽然方向不同，但用户感知差异缩小 |
| **PMF 验证** | 竞品分析的核心假设是"用户愿意为 AI待办转化 付费"。如果 V1.0 根本没有这个功能，无法验证这个假设 |
| **市场窗口** | 竞品分析明确说"市场窗口正在收窄，6-12个月内会有更多 AI+灵感产品"。推迟差异化上线加大了被抢先的风险 |
| **种子用户留存** | 如果 V1.0 只是一个"有AI标签的flomo"，种子用户可能产生"就这？"的印象，D30留存率目标12%面临压力 |

### 建议

**方案 A（推荐）**：将 P1-01（灵感→待办手动转化）提升到 P0。这是成本最低的差异化表达——不需要 AI 批量提取，只需要"点击一条灵感→转为待办"的基本闭环。复杂度低，但 P0 就有了差异化。

**方案 B**：如果坚持当前分期，必须在 PRD 中写清楚**正式的决策理由**，并补充"V1.0 期间如何不让用户觉得只是另一个笔记工具"的体验策略。

**方案 C**：P0 加一个"待办入口占位"——灵感详情页底部有一个"转为待办（即将上线）"的引导，积累期待并收集点击数据验证需求强度。

## 三、🔴 关键问题 2：缺少核心假设验证机制

### 问题描述

竞品分析第 9 节「待确认/假设」中明确标注：

> **核心假设**：产品经理/独立开发者的灵感记录需求是高频刚需（日均5次+）—— **这是最需要验证的假设**

这是整个产品存在的前提。如果假设不成立，后面的差异化、定价、增长全部坍塌。

PRD 有 KPI 体系（WAI 人均≥8条/周、D1/D7/D30 留存率等），但**缺少结构化的假设验证实验设计**。具体缺失：

- **无"假设成立/不成立"的判断标准**：WAI 人均≥8条/周就是验证成功吗？还是需要其他信号？
- **无 pivot 决策点**：如果 D7 留存率远低于 20%，什么时候决定调整方向？
- **无定性验证手段**：除了定量 KPI，是否需要用户访谈来定性理解"为什么记/为什么不记"？
- **无"僵尸用户"甄别**：WAI 只是计数，无法区分"真是灵感记录"还是"随便打几个字测试"。

### 建议

在 PRD 中增加一个「假设验证计划」附录，至少包含：

```
假设1: 灵感记录是高频行为
├─ 验证指标: WAI ≥5条/周（保守阈值）
├─ 验证周期: V1.0 上线后 4 周
├─ 辅助信号: 自然日记录频率分布（是否有 50%+ 用户日均≥1次）
├─ 判断: WAI<3 → 假设不成立，需 pivot
└─ 定性补充: 第3周对10名活跃用户做简短回访

假设2: 用户愿意为 AI待办转化 付费
├─ 验证指标: P1 功能上线后免费→PRO 转化率 ≥4%
└─ 判断: <2% → 付费价值主张需重新设计
```

## 四、🟡 关键问题 3：竞品响应策略缺失

### 问题描述

竞品分析评估 flomo 推出 AI 待办功能的概率为 **"高"**，影响为 **"中"**。这是整个产品面临的最紧迫竞争威胁。

但 PRD 没有设计**时间窗口策略**或**抢先动作**。当前 P0-P1 分期的节奏是：
- V1.0（6-8周后）：无差异化（纯记录+AI标签）
- V1.1（再 4-6周后）：差异化上线（待办+方案）

这意味着**差异化功能需要 10-14 周才上线**。同期 flomo 如果推出类似功能（他们有 200 万用户 + 成熟的技术栈，很可能更快），Inspo 的时间窗口就直接关闭。

### 建议

1. 在 PRD 中增加「竞品监控」章节，定义 flomo/闪念贝壳的监控频率和关键信号
2. 考虑设立**加速路径**：如果竞品监控发现威胁信号，P1 的待办功能可以压缩到 2 周紧急上线
3. 确保 Prompt 和产品逻辑的深度差异化——flomo 即使做待办，也不会有"PM 视角的方案生成"（6段式方案卡），这是深层护城河，需要在 P0 阶段就通过内容营销/种子用户口碑传递出去

## 五、🟢 细节问题

### 5.1 AI 对话入口（P0-10）与方案生成（P1-03）功能重叠

P0-10 灵感详情页 AI 对话的示例能力："可行性分析、竞品推荐、功能拆解"——这与 P1-03 方案生成的 6 段式输出高度重叠。

**建议**：明确区分两个功能的定位：
- 灵感详情页 AI 对话（P0-10）：短对话、碎片化、以单条灵感为上下文，一次对话=1次额度
- 方案生成（P1-03）：结构化长文、以多条灵感为上下文、一次性输出完整方案，消耗 3 次额度

### 5.2 一级标签 7 大类的定义缺失

PRD 多处提到 7 个一级标签（P2-03），但未在任何地方列出这 7 个分类。这影响 AI 标签 Prompt 的设计和用户预期管理。

**建议**：在 PRD 中列出预设的 7 个一级标签及其子标签框架。

### 5.3 拍照无 OCR 的用户体验风险

竞品分析明确说 P0 包含"拍照记录+OCR"。PRD 砍掉 OCR，用户拍照后必须手动打字描述。对设计师用户（US4）来说，这意味着"拍照→打字描述→AI标签"—流程加长。

**建议**：在 V1.0 期间收集"拍照记录的文本描述率"数据——如果大量用户只拍照不写描述，说明 OCR 需求是真实的，应在 V1.1 中恢复。

### 5.4 免费 30 次 AI 额度可能过高

竞品分析定价表中免费版是 **10 次/月**，PRD 提升到 **30 次/月**。虽然成本极低（千人月成本 ¥22），但存在付费用力风险：

- 30 次 = 日均 1 次，对轻度用户足够了
- 如果用户日均记录 1-2 条，30 次刚好覆盖，没有"不够用"的紧迫感
- 竞品分析的 10 次设计意图是"让你体验 AI 价值，但不够用，驱动升级"

**建议**：保持 30 次但加强"用完后的体验落差设计"——目前的四级降级策略第 2 级"预览诱惑"需要真正有诱惑力。或者考虑 15-20 次的中间值。

### 5.5 微信登录优先的合规风险

PRD Q1 决策为"微信授权登录优先"。但竞品分析提到"H5 环境下 Web Speech API 的兼容性问题（特别是 iOS Safari 和 Android 微信内置浏览器）"。

**如果用户在微信内置浏览器打开 H5**，微信授权登录是自然的；**如果用户在系统浏览器（Safari/Chrome）打开**，微信登录需要唤起微信 App，体验会变差。

**建议**：增加"浏览器环境检测→登录方式适配"的逻辑说明。

### 5.6 微信服务号集成从路线图中消失

竞品分析 P2 明确列出"微信服务号集成"（生态获客），但 PRD 的 P2 需求池和 Non-goals 中均未提及。如果未来要转微信小程序，服务号是重要的用户触达通道。

**建议**：在 P2 或未来路线图中保留服务号集成这一项。

### 5.7 数据埋点方案缺失

PRD 列出了大量 KPI（D1/D7/D30 留存率、转化率、WAI），但没有说明这些数据怎么采集和上报。竞品分析提到"数据统计面板"是 P1 功能，但那是面向用户的，不是面向产品团队的。

**建议**：在附录中增加埋点事件清单（至少 10-15 个核心事件），确保关键 KPI 可计算。

## 六、PRD 的亮点（值得保留的）

评审不只是挑问题，这些地方做得很好：

| 亮点 | 说明 |
|------|------|
| **成本模型扎实** | DeepSeek V4 Flash 单次标签成本 ¥0.00072 的计算有具体 token 估算，千人月成本 ¥22，数据靠谱 |
| **模型切换预案** | 三场景四模型的优先级+切换条件+灰度策略，工程上可落地，比大多数 PRD 的"用 API 调一下"强得多 |
| **四级降级策略** | 从智能排队→预览诱惑→手动模式→惊喜回馈，比直接弹窗"升级 PRO"好太多。特别是"惊喜回馈"触发于用户即将流失时的设计 |
| **30秒首次闭环** | 新用户体验路径设计精确到秒，有骨架屏、有 AI 结果动画、有注册引导时机 |
| **火焰 Streak + 冻结卡** | 游戏化设计有具体机制（7 级火焰、损失厌恶），不只是"加个连续签到"的空话 |
| **Non-goals 明确且自洽** | 不做 Demo 生成、不做图片 AI、不做公开社区——这些与竞品分析的定位一致，防止范围蔓延 |
| **用户故事覆盖完整** | 5 个用户故事覆盖了核心场景（记录/方案生成/付费转化/视觉灵感/跨端），比竞品分析的 3 个场景更丰富 |

## 七、综合评分

| 维度 | 评分 | 说明 |
|------|:---:|------|
| **与竞品分析对齐度** | ⭐⭐⭐ | P0/P1 重分配导致核心差异化推迟，降低了与竞品分析策略的一致性 |
| **功能细节完整度** | ⭐⭐⭐⭐⭐ | 验收标准、流程设计、成本模型都非常详细 |
| **市场假设验证设计** | ⭐⭐ | 有 KPI 无实验，这是最大的结构性缺失 |
| **竞品防御意识** | ⭐⭐ | 未体现时间窗口紧迫感和竞品响应策略 |
| **工程可落地性** | ⭐⭐⭐⭐⭐ | 模型切换预案、API 对接细节、存储架构都很务实 |
| **用户体验设计** | ⭐⭐⭐⭐ | 四级降级、Streak、30秒闭环等设计有深度，但拍照无 OCR 是遗憾 |
| **Non-goals 纪律** | ⭐⭐⭐⭐⭐ | 砍得很果断，方向清晰 |

**综合**：⭐⭐⭐⭐ — 一份质量不错的 PRD，但在策略层面有三个需要在 V1.0 前修正的问题。
