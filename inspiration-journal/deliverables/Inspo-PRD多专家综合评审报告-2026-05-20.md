## 三、最关键的三个决策建议

### 决策 1：P1-01（灵感→待办手动转化）是否提到 P0？

> **4 位专家中 3 位认为这是当前 PRD 最大的策略问题。**

| 支持提到 P0 的理由 | 反对/保留的理由 |
|-------------------|----------------|
| V1.0 否则只是"有 AI 标签的 flomo"，差异化太薄 | P0 已有 AI 对话作为轻量分析入口 |
| 首次体验第 5 步承诺了待办转化但无法兑现 | 待办管理(P1-06)牵一发动全身 |
| 手动转化实现成本极低（一张编辑面板+DB），1-2 天 | 等 V1.1 集中上线待办模块更完整 |
| flomo 4-6 周内可能上线 AI 标签，抹平 V1.0 差异化 | — |

**建议**：提到 P0。最低成本方案——只做「灵感详情页 → 点击"转为待办" → 弹出编辑面板预填标题/详情 → 保存」，附带一个极简待办列表（勾选=完成）。不需要 AI 批量提取（P1-02）、不需要方案生成（P1-03）、不需要日历（P1-07）。这样 V1.0 就兑现了"记录→AI 分析→转化为行动"的完整闭环。

### 决策 2：V1.0 的用户价值主张是否需要调整？

> PM 许清楚指出："PRD 说'AI 帮你实现'，V1.0 实际做的是'AI 帮你整理'。"

如果决策 1 通过（待办提 P0），主张可维持"抓住灵感，AI 帮你实现"。
如果决策 1 不通过，建议 V1.0 阶段调整为"抓住灵感，AI 帮你整理"，V1.1 升级。

### 决策 3：PRD 是否需要在 V1.2 中加入假设验证计划？

> **必不可少。** 这是 PM 和 QA 的共识。没有假设验证计划的 MVP PRD 只能指导「做什么功能」，不能指导「怎么判断功能做对了没」。

建议在 PRD 附录中加入：
- 核心假设 4 条（日均灵感≥5 次、愿付费¥99、AI 标签准确率≥80%、方案 NPS≥30）
- 每条假设的验证窗口（4 周/8 周）
- Pivot 触发条件（定量阈值 + 定性信号）
- 验证手段（定量埋点 + 5-8 名用户回访）

## 四、修订后的 P0 时间线建议（含阻塞项解决）

```
Week 1-2: 阻塞项清零 + 基础架构
├─ B7 CodingPlan API 实测（100 样本 × 1000 调用）
├─ B8 Web Speech API 6 组合兼容性测试
├─ B1 微信登录方案决策（手机号为主 / 微信扫码 / 引导微信内打开）
├─ B6 Supabase RLS 启用 + API Key 隔离确认
├─ B2 离线同步 LWW 策略确定
├─ Supabase 数据模型 + Dexie 表定义
└─ PWA 脚手架 + 代码仓库初始化

Week 3-4: P0 核心功能 v1
├─ P0-01 文字极速记录
├─ P0-02 语音记录（含取消手势、ASR 降级、转写失败状态）
├─ P0-03 拍照记录（含压缩+上传进度）
├─ P0-07 用户注册登录（含浏览器环境检测+自适应登录方式）
└─ P0-05 AI 标签 Pipeline（含超时/失败状态机）

Week 5-6: P0 核心功能 v2 + 质量加固
├─ P0-04 离线同步引擎（含冲突处理、配额管理、网络切换保护）
├─ P0-08 灵感列表+标签筛选+搜索
├─ P0-09 AI 额度系统（含原子扣减、四级降级触发条件）
├─ P0-06 PWA 安装引导+离线缓存优化
├─ 【新增】→ 待办手动转化（P1-01 提到 P0）+ 极简待办列表
└─ 内部 Dogfooding 1 周

Week 7-8: 上线准备 + 假设验证启动
├─ P0-10 AI 对话（含 SSE 流式确认/降级）
├─ 埋点上线（≥15 个核心事件）
├─ 假设验证计划启动
├─ 灰度发布（种子 50 → 200 → 全量）
└─ Bug 修复 + 性能调优
```
