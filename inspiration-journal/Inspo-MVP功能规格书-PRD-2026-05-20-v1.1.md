## 一、产品目标

Inspo 的三组正交目标，分别对应体验、价值和增长：

| # | 目标 | 衡量指标 | 目标值（MVP 3个月） |
|---|------|----------|---------------------|
| G1 | **极速捕获** — 让用户在任何场景下3秒内开始记录灵感 | 记录启动耗时 P50 | <1.5秒（PWA热启动） |
| G2 | **AI增强闭环** — 灵感记下后自动产生可行动产出 | 灵感→AI处理完成率 | ≥90%（含标签/标题/摘要） |
| G3 | **从灵感到执行** — 用户将灵感转化为待办或方案 | 灵感→待办/方案转化率 | ≥25%（活跃用户中） |

三组目标的关系：G1 是入口（没有记录就没有一切），G2 制造"记了就有价值"的正循环，G3 是付费转化的核心驱动力。V1.0 即可验证 G3——手动待办转化功能已纳入 P0，用户记录灵感→AI 自动标签→手动转为待办，形成最小行动闭环。

## 二、用户故事

### US1：产品经理的灵感闪念
> "开会时突然想到一个功能点子，拿出手机2秒内开始打字，写完就走，不用填标题、不用选分类。半小时后再打开，AI 已经帮我打好了 #产品 标签，还生成了一个标题。"

**场景**：会议间隙，单手操作，10-30秒完成一次记录。

### US2：独立开发者的周日晚间整理
> "一周积累了20多条零散想法。周日晚8点收到推送，打开App看到本周积累了丰富的灵感。进入灵感列表，把最有价值的5条灵感逐条转为待办——设置优先级和截止日期。虽然方案生成要等V1.1，但现在就能把想法变成行动清单。"

**场景**：深度使用，灵感回顾→手动待办转化全链路。"生成方案"是P1功能，但"转为待办"P0就有。

### US3：免费用户探索付费
> "这个月30次免费AI额度用了28次。我在灵感详情页点开AI对话，和AI就一条灵感聊了10分钟——它帮我分析了可行性，还推荐了2个竞品。对话中AI建议的行动项，我直接一键转为待办。这种深度体验让我决定升级 PRO。"

**场景**：免费额度充裕（30次/月）→ AI对话体验价值感知 + 对话中一键转待办 → 付费转化。

### US4：设计师的视觉灵感
> "看到一张好的交互设计，拍照记录，写了一句描述。AI 自动打了 #UX 标签。后来我回顾时，可以按标签快速筛选所有 UX 相关灵感。"

**场景**：拍照记录→手动写描述→AI标签→按需检索。（图片AI理解MVP不做）

### US5：多设备接力记录
> "地铁上用手机语音记录了一段想法（60秒内免费），到公司后在电脑上打开 PWA 看到转写结果和 AI 标签都已完成。继续编辑完善。"

**场景**：手机语音输入→云端ASR转写→PWA跨端同步。

### US6：清理低价值灵感

### 3.1 目标用户画像

| 维度 | 产品经理 | 独立开发者 |
|------|---------|-----------|
| 日均灵感数 | 8-15条 | 3-8条 |
| 记录场景 | 会议间隙、通勤、睡前 | 开发中、阅读时、深夜 |
| 主要记录方式 | 文字为主（80%），偶尔语音 | 文字（60%）+ 代码片段（20%）+ 链接（20%） |
| 最大痛点 | 记了找不到、记了不行动 | 想法散落各处、无法串联成方案 |
| 付费意愿 | 中等（公司报销可能性高） | 高（自己说了算，ROI敏感） |

### 3.2 关键UX原则

- **拇指区优先**：核心操作区占据屏幕底部70%，适配单手操作
- **零摩擦记录**：打开即记，无需任何前置操作（选分类/填标题/打标签）
- **渐进式AI呈现**：AI标签和摘要不是记录的前提，而是记录的结果——后台异步处理，用户无感知
- **30秒首次闭环**：新用户从打开到看到AI处理结果 ≤30秒
- **火焰 Streak 激励**：7级火焰视觉（灵感日历连续记录天数），利用损失厌恶驱动留存
- **灵感冻结卡**：连续3天未记录可触发，赠送1次方案生成额度，拉回沉睡用户

### 3.3 免费版AI额度分配策略

免费版每月 **30次** AI额度（记录一条灵感→生成标签/标题/摘要 = 消耗1次）。成本模型：

| 模型 | 单次标签成本 | 30次/月/人 | 千人月成本 | 万人月成本 |
|------|------------|-----------|-----------|-----------|
| DeepSeek V4 Flash（主） | ¥0.00072 | ¥0.022 | **¥22** | **¥216** |
| 豆包Lite（备选） | ¥0.0002 | ¥0.006 | ¥6 | ¥60 |

30次足够覆盖日均1次的标签生成需求，无需"5-2-2-1"紧张分配策略。

**用完后的四级平滑降级**（禁止粗暴弹窗）：
1. **智能排队**：剩余额度>0但<正常处理阈值时，AI处理进入低优先级队列（仍处理但延迟<5分钟）
2. **预览诱惑**：额度=0时，展示AI标签/标题的前几个字，完整内容模糊+"升级PRO解锁"
3. **手动模式**：额度=0持续3天，引导手动标签
4. **惊喜回馈**：连续5天未活跃+额度=0+注册>14天→随机赠送1-3次额度（每月≤1次）

**flomo 的核心盲区**（Inspo 的差异化窗口）：
1. flomo 的标签是手动的、纯文本的——无 AI 自动分类
2. flomo 的连接是手动打双链——无 AI 自动关联/聚合
3. flomo 不提供任何"下一步行动"——记了就完了，缺乏"灵感→执行"闭环
4. flomo 的回顾是随机漫步——无 AI 方案生成

## 五、数据依据

### 5.1 AI API选型及成本模型

**AI能力提供商**：**火山方舟 CodingPlan API**（用户提供的URL+API Key），支持多模型切换（DeepSeek V4 Flash、豆包系列、Kimi、GLM等）。

**标签生成场景（场景A）— DeepSeek V4 Flash**：

| 参数 | 数值 |
|------|------|
| 模型 | DeepSeek V4 Flash（通过CodingPlan调用） |
| 输入（系统prompt 200 + 用户125字×1.8） | ~425 tokens |
| 输出（标签≤5个，均值3个，~135 tokens） | ~135 tokens |
| 输入价格 | $0.14 / 1M tokens |
| 输出价格 | $0.28 / 1M tokens |
| **单次标签成本** | **≈ $0.0001 ≈ ¥0.00072** |

> ⚠️ 通过火山引擎 CodingPlan 调用可能有渠道加价，预估加价50-100%后单次成本仍在¥0.001-0.0015区间。千人月成本<¥45。

**四类AI场景汇总**：

| 场景 | 模型建议 | 消耗倍数 | 单次成本（估算） | 说明 |
|------|---------|---------|-----------------|------|
| A — 标签+标题+摘要 | DeepSeek V4 Flash | 1x | ¥0.00072 | 轻量分类任务 |
| B — 灵感分析+待办提取 | DeepSeek V4 Flash / 豆包Pro | 1x | ¥0.001-0.008 | 中等复杂度 |
| C — 方案生成 | DeepSeek V4 Pro / 豆包Pro | **3x** | ¥0.035-0.047 | 长文本生成 |
| D — AI对话 | DeepSeek V4 Flash / 豆包Pro | 1x/轮 | ¥0.002-0.016 | 1轮=用户1条消息+AI 1条回复 |

> **注**：V1.0的待办手动转化不消耗AI额度（零AI成本），仅AI批量提取待办（P1）和AI方案生成（P1）消耗额度。

    │         └─ 上滑到取消区域（屏幕上方1/3）→ 松手 → 取消录音

## 八、Non-goals（明确不做）

以下功能**明确排除**在 MVP 及近期路线图中：

| 不做 | 原因 |
|------|------|
| ❌ Demo/原型生成 | 产品定位是"灵感到方案"而非"方案到Demo"。跨到代码生成会模糊核心价值且大幅增加AI成本 |
| ❌ 图片AI理解（OCR/描述生成） | MVP仅保存照片+用户手动描述。多模态模型成本高且当前能力不足以准确理解灵感类图片，V1.1重新评估 |
| ❌ 原生App（iOS/Android） | MVP验证用PWA，验证PMF后再考虑原生。PWA已覆盖离线、推送、安装到桌面等核心能力 |
| ❌ 多人协作/团队版 | 个人工具先行，协作是规模化后的方向 |
| ❌ 公开灵感社区/广场 | 灵感是私密的，社区功能与产品气质不符 |
| ❌ 自定义AI Prompt模板 | 初期固定Prompt确保输出质量一致，V2可开放 |
| ❌ 第三方集成（飞书/Notion/Slack） | MVP聚焦核心闭环，集成在P2规划 |
| ❌ AI生成代码/设计稿 | 超出"方案"范畴，属于另一个产品 |
| ❌ Web端独立域名（桌面优先） | MVP以移动端H5+PWA为主，桌面端仅保证基础可用 |
| ❌ 用户自定义标签体系 | MVP使用固定7大类+子标签预设，不让用户从零建标签体系 |
| ❌ 灵感公开分享/社交功能 | 灵感私密性第一，分享仅限方案导出 |
| ❌ Redis | MVP用Supabase内置能力即可满足，不过度工程 |
| ❌ CRDT同步 | MVP用Last-Write-Wins(LWW)方案，简化实现 |
| ❌ 自建数据分析后台 | 用Supabase Dashboard + Umami自部署满足报表需求，暂不自建 |
