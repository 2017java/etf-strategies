# 配料表解读 Skill 设计文档（Coze，大陆用户，v1）

## 1. 概述

本项目目标是在 Coze 平台交付一个可直接使用的配料表解读能力。用户输入食品/饮料配料信息（图片或文本）后，系统输出红黄绿风险卡片与可展开详情，帮助用户快速理解当前食物/饮料的潜在健康风险。

本设计基于以下已确认决策：

- 架构：`Bot + Workflow + 技能/插件`
- 输入方式：`拍照/OCR + 文本粘贴`
- 风险框架：`通用风险库优先，后续扩展特殊人群`
- 输出形态：`卡片摘要 + 详情展开`
- 首发范围：`包装食品与饮料`
- 标红策略：`严格标红（高置信命中才红）`
- 详情展示：`命中点 + 简短原因`
- 数据策略：`默认不长期保存输入数据`
- 首发渠道：`Coze 内测链接/私域群`
- 核心指标：`有用反馈率 + 7天二次使用率`

## 2. 目标与非目标

### 2.1 目标（v1）

- 支持用户通过图片或文本提交配料信息。
- 在可接受时延内返回红黄绿结果与简要解释。
- 提供“查看详情”能力，解释触发原因。
- 在文案和输出上满足大陆用户理解习惯与基础合规要求。

### 2.2 非目标（v1 不做）

- 不做医疗诊断建议，不输出疾病治疗类结论。
- 不做个性化人群模型（孕妇、糖尿病、儿童等）判定。
- 不做用户历史饮食记录与长期画像。
- 不做全品类扩展（如散装食品、餐馆菜品成分估算）。

## 3. 总体架构

### 3.1 三层结构

1. Bot 层（交互层）
- 接收用户图片/文本。
- 展示卡片摘要、详情按钮、失败回退提示。
- 收集“有帮助/没帮助”等反馈信号。

2. Workflow 层（编排层）
- 控制全流程节点执行顺序与异常回退。
- 统一结构化输出（风险等级、命中项、解释）。
- 管理置信度与结果可解释信息。

3. 技能/插件层（能力层）
- OCR/视觉解析能力（优先平台内置能力）。
- 成分标准化能力（同义词归一、分词清洗）。
- 风险规则匹配与打分能力。
- 解释生成能力（简短、可读、不过度专业化）。

### 3.2 推荐部署形态

- 对外主入口：1 个独立 Bot。
- Bot 内调用：1 条主 Workflow。
- 可复用能力：拆成多个技能/插件，供后续其他 Bot 复用。

## 4. Workflow 设计

### 4.1 节点清单

#### 节点 A：输入校验

输入：
- `user_input_text`（可空）
- `user_input_image`（可空）

规则：
- 二者至少一个存在，否则返回引导文案：
  - “请上传配料表图片，或直接粘贴配料文本。”

输出：
- `input_mode`：`image` / `text` / `both`

#### 节点 B：OCR 提取（仅图片触发）

输入：
- `user_input_image`

处理：
- 调用 Coze 内置视觉/OCR 能力提取文本。
- 同时生成 `ocr_confidence`（无原生置信度时由启发式规则估算）。

输出：
- `ocr_text`
- `ocr_confidence`

失败回退：
- 返回提示：
  - “图片中配料信息不清晰，请补拍更清晰的配料表（正对拍摄，避免反光）。”

#### 节点 C：文本归一与成分切分

输入：
- `raw_text = user_input_text || ocr_text`

处理：
- 去除广告词、无关文案与乱码。
- 统一标点与分隔符（逗号、顿号、分号、括号）。
- 成分同义词归一（示例：白砂糖/蔗糖 -> 添加糖标签）。
- 产出结构化成分数组。

输出：
- `ingredient_list[]`
- `normalize_confidence`

#### 节点 D：规则匹配与打分

输入：
- `ingredient_list[]`
- `ocr_confidence`
- `normalize_confidence`

处理：
- 命中规则库并累计得分。
- 记录命中证据（命中词、所属规则、简短原因）。
- 计算总体风险等级。

输出：
- `risk_level`：`red | yellow | green`
- `score`
- `hits[]`
- `decision_confidence`

#### 节点 E：结果生成与渲染数据

输入：
- `risk_level`
- `hits[]`

处理：
- 生成卡片摘要文案（默认短文本）。
- 生成可展开详情（命中点 + 简短原因）。
- 追加固定免责声明。

输出：
- `summary_card`
- `detail_items[]`
- `disclaimer`

## 5. 风险分级策略（严格标红）

### 5.1 标红规则（v1）

仅当以下条件满足时标记红色：

- 命中至少 1 条高风险硬规则；
- 且 `decision_confidence >= 0.8`。

否则：

- 命中中风险或提示性规则 -> 黄色；
- 未命中风险规则 -> 绿色。

### 5.2 规则库结构

每条规则建议包含：

- `rule_id`
- `name`
- `match_keywords[]`
- `risk_weight`
- `severity`（high/medium/low）
- `rule_type`（hard/soft）
- `short_reason`
- `source_type`（国标/官方指南/通用建议）
- `enabled`

### 5.3 规则规模建议（v1）

- 首批维护 30-50 条高频规则。
- 优先覆盖：高糖、高钠、咖啡因提示、争议添加剂高频词。

## 6. 用户交互设计

### 6.1 首轮引导

- “请拍摄配料表，或直接粘贴配料文本，我来帮你解读。”

### 6.2 结果样式

- 默认：红黄绿卡片 + 一句话结论 + 前 3 个命中点。
- 详情：点击“查看详情”后展示命中点与简短原因。

### 6.3 错误提示

- 图片不可读：引导重新拍摄。
- 文本太短或无配料结构：提示补充更完整配料表。

### 6.4 合规文案

固定免责声明：

- “本结果仅供营养健康参考，不构成医疗建议。”

## 7. 数据与合规策略

- 默认不长期保存原始图片和原始文本。
- 会话结束后仅保留必要匿名统计指标（用于效果评估）。
- 输出文案避免使用诊断/治疗类措辞。
- 外部 API 若后续接入，仅用于补全数据，不直接决定医疗相关结论。

## 8. 指标与运营观察

### 8.1 北极星指标

1. 有用反馈率（用户点击“有帮助”比例）
2. 7 天二次使用率

### 8.2 辅助指标

- OCR 成功率
- 首次响应时延
- 详情展开率
- 红黄绿分布占比（用于监控规则过严或过松）

## 9. 测试计划与验收标准

### 9.1 功能测试

- 图片输入可返回有效卡片结果。
- 文本输入可返回同结构结果。
- 点击“查看详情”可查看命中点与简短原因。
- 无码或空输入时给出正确引导。
- OCR 失败时给出可执行重试建议。

### 9.2 策略测试

- 高风险硬规则 + 高置信度命中 -> 红色。
- 相同命中但低置信度 -> 不得红色（降级黄）。
- 无风险命中 -> 绿色。

### 9.3 合规测试

- 每次结果均包含免责声明。
- 不出现医疗诊断型措辞。
- 不持久化原始图片与文本。

### 9.4 验收通过条件

- 核心链路成功率 >= 95%（内测样本）。
- 结果结构稳定，无空字段或崩溃分支。
- 至少完成 30 个真实样本回归验证，误报率可接受。

## 10. 迭代路线（v1 -> v2）

v2 可扩展方向：

- 特殊人群策略包（孕妇/儿童/慢病）。
- 条码识别 + 商品库补全。
- 用户授权后的饮食记录与周期建议。
- 规则库运营后台（审核、版本、回滚）。
